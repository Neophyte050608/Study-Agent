# 项目异常保底策略

本文基于当前项目代码，对“项目各个部分出错时有什么保底策略”做一次系统梳理。这里的“保底策略”不是只指 try-catch，而是指在异常、超时、依赖不可用、结果不可信、链路中断、输入非法等情况下，系统如何：

- 不直接崩
- 给用户可理解的反馈
- 给上层稳定的数据结构
- 给运维留下可排查的痕迹
- 尽量把失败收敛在局部，而不是向全链路扩散

从整体设计看，这个项目的保底策略主要分为 8 层：

1. 入口层保底：统一异常返回、限流、参数清洗
2. 路由层保底：前置规则命中、意图树回退、澄清而不是误判
3. 模型层保底：候选切换、首包探测、熔断、fallback 模型
4. RAG 层保底：检索空结果兜底、Web fallback、评估 JSON 降级
5. MCP 层保底：参数归一化失败兜底、超时重试、stub 降级、审计记录
6. 流式输出层保底：取消、失败收口、主通道优先、旁路观察者不影响主链路
7. 任务/IM/入库层保底：异步异常隔离、会话状态清理、任务快照和失败摘要
8. 观测层保底：失败可追踪、状态可回放、审计可复盘

下面按模块展开。

---

## 1. 入口层保底

### 1.1 全局异常兜底

- 关键类：
  - `src/main/java/com/example/interview/controller/GlobalExceptionHandler.java`

当前项目在 Controller 层做了统一异常收口，避免把堆栈直接暴露给前端。

#### 已有保底行为

1. 上传超过限制时，统一返回 413
   - 普通 HTTP：`{"message":"上传文件超过服务端限制，请上传 20MB 以内 PDF"}`
   - SSE：直接返回纯文本，避免流式端解析异常
2. 参数非法时，统一返回 400
   - 普通 HTTP：JSON
   - SSE：纯文本
3. 未捕获异常时，统一返回 500
   - 对用户暴露的是稳定话术：`服务器内部错误，请稍后重试`

#### 价值

- 避免接口返回风格混乱
- 避免敏感错误信息外泄
- SSE 和普通接口分别做了适配，不会因为统一 JSON 破坏流式消费

#### 当前结论

入口层已经具备“统一失败出口”的保底能力，这是线上系统非常关键的一层。

---

### 1.2 限流保底

- 关键类：
  - `src/main/java/com/example/interview/security/RateLimitInterceptor.java`

#### 已有保底行为

- 按 IP 做分钟级滑动窗口限流
- 超限时直接返回 429
- 响应体固定为：
  - `{"error":"请求过于频繁，请稍后再试"}`

#### 价值

- 防止某个用户或脚本打爆服务
- 在大模型和检索服务比较贵时，限流本身就是保底策略

#### 风险点

- 当前是单机内存计数，不是分布式共享
- 服务重启后计数丢失

#### 优化建议

- 上 Redis 做分布式限流
- 针对 `/stream`、`/upload`、`/eval` 做分级限流

---

### 1.3 输入清洗保底

- 关键类：
  - `src/main/java/com/example/interview/service/ChatStreamingService.java`
  - `src/main/java/com/example/interview/security/InputSanitizer.java`

#### 已有保底行为

- 流式聊天入口会先对用户输入做 `sanitize`
- 目的是在链路最前面消减非法字符、脏输入、潜在注入干扰

#### 价值

- 避免脏输入把下游 prompt、日志、会话存储污染掉
- 这是“入口前置防御”的典型做法

---

## 2. 意图路由层保底

这一层是“别把用户问错路由了”。  
对话系统里，错误路由往往比直接失败更危险，因为它会输出“看起来正常但其实答非所问”的内容。

---

### 2.1 前置规则保底

- 关键类：
  - `src/main/java/com/example/interview/intent/IntentPreFilter.java`

#### 已有保底行为

1. `/help` 直接回复帮助文案
2. `/clear` 直接清上下文，不走模型
3. 简单问候语直接回复，不浪费模型调用
4. 明确的“生成报告”“开始面试”“刷题”“查画像”直接路由
5. 即使没识别到明确任务，也会先识别业务域：
   - `INTERVIEW`
   - `CODING`
   - `KNOWLEDGE`
   - `PROFILE`

#### 这层的保底本质

先用确定性规则截住最稳定的意图，减少：

- 大模型误判
- 模糊输入乱跳转
- 成本浪费
- 首问冷启动时间过长

#### 典型收益

“你好”“/help”“/clear”“来一道算法题”这类高确定性输入，不需要经过复杂推理链。

---

### 2.2 意图树回退保底

- 关键类：
  - `src/main/java/com/example/interview/service/IntentTreeRoutingService.java`
  - `src/main/java/com/example/interview/intent/IntentRoutingDecision.java`
  - `src/main/java/com/example/interview/agent/TaskRouterAgent.java`

#### 已有保底行为

1. Query 为空时直接 `fallback()`
2. 域分类不稳定时，回退到全局叶子分类
3. 域下没有叶子时，回退旧链路
4. 置信度低时，不强判，转“澄清”
5. Top1/Top2 分差太小，不强判，转“澄清”
6. `UNKNOWN` 且不适合继续分类时，返回引导话术
7. 意图树不可用时，回退到旧版 ReAct 路由 `reactDispatch`

#### 这里最重要的保底思想

不是“非要给出一个答案”，而是：

- 宁可澄清
- 宁可回退旧链路
- 宁可提示用户重新表达
- 也不要高置信度误判

这比“瞎猜一个 taskType”安全得多。

---

### 2.3 ReAct 路由兜底

- 关键类：
  - `src/main/java/com/example/interview/agent/TaskRouterAgent.java`

#### 已有保底行为

当意图树不可用或要求回退时，`TaskRouterAgent` 会走 `reactDispatch`：

1. 构造 prompt 调模型判任务
2. 首包探测失败时，使用固定 JSON 兜底：
   - `{"taskType":"UNKNOWN"}`
3. 若最终仍无法识别，返回统一引导语
4. 若识别出 taskType，再重新封装成 `TaskRequest` 继续走正式链路

#### 价值

- 路由系统不会因为新意图树异常而整体瘫痪
- 老链路还能托底

#### 当前结论

意图层已经形成了“三层保底”：

1. 规则前置
2. 意图树分类
3. ReAct 旧链路回退

这是比较完整的路由容错结构。

---

## 3. 模型路由层保底

这是项目里保底做得比较完整的一层。

---

### 3.1 候选模型切换保底

- 关键类：
  - `src/main/java/com/example/interview/modelrouting/ModelRoutingExecutor.java`
  - `src/main/java/com/example/interview/modelrouting/RoutingExecutionTemplate.java`
  - `src/main/java/com/example/interview/modelrouting/ModelSelector.java`

#### 已有保底行为

1. 候选列表为空时直接失败，不继续误调
2. 候选列表存在时，按优先级依次尝试
3. 当前候选失败，则切下一个候选
4. 所有候选都失败，才整体抛 `ModelRoutingException`
5. `RoutingExecutionTemplate` 可指定 fallbackSupplier，最后统一降级

#### 保底效果

不是“一个模型挂了，全业务挂”，而是“这个模型挂了，换下一个候选继续跑”。

---

### 3.2 fallback 模型保底

- 关键类：
  - `src/main/java/com/example/interview/modelrouting/RoutingChatService.java`

#### 已有保底行为

`RoutingChatService` 中存在统一的 `fallbackChatModel`，并且多个调用方法都支持：

- 路由候选正常则走候选
- 路由关闭/候选空/候选失败则走 fallback model

典型方法：

- `call`
- `callWithMetadata`
- `callStream`
- `callWithFirstPacketProbeSupplier`
- `callSupplier`

#### 价值

- 即使动态模型配置、候选模型、路由规则出错，也仍有最后一个默认模型兜底

---

### 3.3 首包超时保底

- 关键类：
  - `src/main/java/com/example/interview/modelrouting/RoutingChatService.java`
  - `src/main/java/com/example/interview/modelrouting/FirstTokenProbeInvoker.java`

#### 已有保底行为

对于高实时性场景，项目不是只看“HTTP 成功没成功”，而是看“首 token 有没有及时返回”。

如果首包探测失败：

1. 统计 timeout/failure 次数
2. 记录 warn 日志
3. 执行统一 fallbackSupplier

#### 价值

这能处理 LLM 特有的“连接成功但一直不出 token”的假死问题，比普通 HTTP 重试更贴近真实线上故障。

---

### 3.4 熔断保底

- 关键类：
  - `src/main/java/com/example/interview/modelrouting/ModelHealthStore.java`
  - `src/main/java/com/example/interview/modelrouting/ModelHealthProbeScheduler.java`

#### 已有保底行为

项目实现了三态熔断：

1. `CLOSED`
   - 正常放行
2. `OPEN`
   - 达到失败阈值后熔断
   - 冷却时间内直接拒绝，不再继续打这个模型
3. `HALF_OPEN`
   - 冷却后少量放行做探活
   - 成功则恢复
   - 失败则重新打开熔断

并且有定时任务定期探测主模型健康。

#### 价值

- 故障模型不会被持续打挂
- 系统能自动恢复
- 对 LLM 场景尤其有效

#### 当前结论

模型层保底已经覆盖：

- 多候选切换
- fallback 模型
- 首包探测
- 熔断与自动恢复

这是这个项目稳定性的核心支柱之一。

---

## 4. RAG 与知识检索层保底

这一层的核心问题是：  
本地检索召不出来、召回质量差、模型评估超时、输出 JSON 不合法时，如何保证对话还能继续。

---

### 4.1 检索通道局部失败保底

- 关键类：
  - `src/main/java/com/example/interview/service/RAGService.java`

#### 已有保底行为

在 `retrieveHybridDocuments` 中，并行执行三路召回：

1. 意图定向检索
2. 向量检索
3. 图谱检索

每一路都各自 catch 异常，失败后返回空列表，而不是把整个检索链路打断。

#### 价值

- 某一个通道挂了，不影响其他通道继续返回结果
- 这是典型的“并行通道局部失败隔离”

---

### 4.2 Web 搜索兜底

- 关键类：
  - `src/main/java/com/example/interview/service/RAGService.java`
  - `src/main/java/com/example/interview/tool/WebSearchTool.java`
  - `src/main/java/com/example/interview/config/RagRetrievalProperties.java`

#### 已有保底行为

当本地检索为空，或检索质量低于阈值时，会触发 Web fallback：

- `NONE`：不启用
- `ON_EMPTY`：本地空结果才启用
- `ON_LOW_QUALITY`：空结果或低质量结果都启用

`WebSearchTool` 自身也做了兜底：

- 输入为空时直接返回空列表
- 请求或解析异常时记录 warn，并返回空列表

#### 价值

- 本地知识库不全时，系统仍然能给出一定回答能力
- Web fallback 是 RAG 层最关键的保底能力之一

#### 风险点

- 外网搜索结果可信度低于本地知识
- 当前没有更细粒度的来源置信控制

#### 优化建议

- 回答中明确区分“本地证据”与“外网兜底证据”
- 对 Web fallback 增加禁用场景，如高风险面试评分场景

---

### 4.3 Query 改写失败保底

- 关键类：
  - `src/main/java/com/example/interview/service/RAGService.java`

#### 已有保底行为

关键词提取/查询改写失败时：

1. 会进行有限重试
2. 若最终失败，直接退回原始问答拼接文本检索
3. `RewrittenQuery.fallback(raw)` 保证调用方拿到的仍是合法结构

#### 价值

- 不会因为 query rewrite 失败导致整个 RAG 不可用

---

### 4.4 评估模型失败保底

- 关键类：
  - `src/main/java/com/example/interview/service/RAGService.java`

#### 已有保底行为

`evaluateWithKnowledge` 和 `processAnswer` 在模型调用失败时，不会直接抛给用户，而是返回降级 JSON：

- `score = 0`
- `accuracy / logic / depth / boundary = 0`
- `deductions = ["服务降级，未产出扣分点"]`
- `citations = []`
- `conflicts = []`
- `feedback` 给出超时或服务不可用话术
- `nextQuestion` 仍给一个保底字段

#### 价值

- 前端永远能拿到结构化评估结果
- 不会因为字段缺失导致页面崩

这是一种很成熟的“结构稳定优先”的保底策略。

---

### 4.5 评估结果修补保底

- 关键类：
  - `src/main/java/com/example/interview/service/RAGService.java`

#### 已有保底行为

`validateEvidenceReferences` 对模型输出做二次修补：

1. 过滤不存在的 citations/conflicts
2. 如果 citations 为空，自动补第一条合法证据
3. 如果模型遗漏 `nextQuestion`，自动给默认追问
4. 如果校验过程本身失败，保留原始结果，不二次打断

#### 价值

- 避免“大模型输出结构对了，但引用不可信”
- 避免字段遗漏影响上游逻辑

---

### 4.6 文本切块保底

- 关键类：
  - `src/main/java/com/example/interview/rag/RecursiveChunker.java`

#### 已有保底行为

递归按自然边界切分，如果所有自然边界都试过仍然过长：

- 启用 `fallbackTokenSplitEnabled` 时
- 退回到 `TokenTextSplitter` 做物理切分

#### 价值

- 超长文档不会因为“切不动”而直接失败
- 这是知识入库和检索稳定性的底层保底

---

## 5. MCP 能力调用层保底

这部分是项目里做得最明显的“工程化降级”之一。

---

### 5.1 参数归一化失败保底

- 关键类：
  - `src/main/java/com/example/interview/service/McpGatewayService.java`
  - `src/main/java/com/example/interview/service/mcp/CapabilityParamNormalizer.java`

#### 已有保底行为

如果 capability 参数归一化失败：

- 不直接抛异常给用户
- 返回统一 fallback payload：
  - `status = fallback`
  - `capability`
  - `attempt`
  - `message`
  - `errorCode`
  - `retryable`
  - `result` 中保留原始 params/context

#### 价值

- 上层仍然能拿到结构化结果
- 审计和排查都有依据

---

### 5.2 网关选择失败保底

- 关键类：
  - `src/main/java/com/example/interview/service/McpGatewayService.java`
  - `src/main/java/com/example/interview/service/mcp/GatewaySelectionStrategy.java`

#### 已有保底行为

如果目标网关没配置好，且 `fallbackToStub=true`：

- 自动回退到 `StubMcpCapabilityGateway`
- `discoverCapabilities` 仍能返回能力列表
- `invoke` 仍能返回模拟成功结果

#### Stub 能做什么

- `obsidian.write` 返回伪 notePath
- `code.execute` 返回 `exitCode=0`
- 其他能力返回：
  - `status=stub`
  - 原始 params

#### 价值

- 开发环境、演示环境、依赖未接通时系统仍可跑通主链路

---

### 5.3 超时重试保底

- 关键类：
  - `src/main/java/com/example/interview/service/mcp/GatewayInvocationExecutor.java`
  - `src/main/java/com/example/interview/service/McpGatewayService.java`

#### 已有保底行为

1. 调用通过 `CompletableFuture.orTimeout` 加超时
2. 按 `retries + 1` 次尝试
3. 仅 retryable 异常继续重试
4. 最终失败时：
   - 能 stub 就 stub
   - 不能 stub 就返回 fallback 结构

#### 价值

- 短暂网络抖动不会直接让请求失败
- 超时、远端异常、协议异常都被统一映射

---

### 5.4 统一错误语义保底

- 关键类：
  - `src/main/java/com/example/interview/tool/McpGatewayException.java`
  - `src/main/java/com/example/interview/tool/FastMcpCapabilityGateway.java`
  - `src/main/java/com/example/interview/tool/McpBridgeCapabilityGateway.java`

#### 已有保底行为

不同远端网关的错误最终统一收敛到：

- `errorCode`
- `retryable`
- `message`

#### 价值

- 上层不需要区分 HTTP 5xx、连接错误、协议错误
- 重试和降级依据统一

---

### 5.5 审计保底

- 关键类：
  - `src/main/java/com/example/interview/service/mcp/GatewayAuditRecorder.java`

#### 已有保底行为

无论成功、fallback、fallback_stub、error，都会记录：

- operator
- action
- status
- errorCode
- retryable
- traceId

#### 价值

- 用户看到的是“服务降级”
- 运维看到的是“哪一步降级、为什么降级”

这就是典型的“用户体验保底 + 运维排障保底”双重设计。

---

## 6. 流式输出层保底

流式输出最怕的是：

- 前端断开
- 中途取消
- 旁路监听器抛异常
- 流输出到一半链路挂掉

这个项目在这层做了比较好的失败收口。

---

### 6.1 主流优先保底

- 关键类：
  - `src/main/java/com/example/interview/stream/ObservableStreamEmitter.java`

#### 已有保底行为

事件发送顺序：

1. 先发给主发送器 `primary`
2. 再发给 observers
3. observers 抛异常时直接忽略

#### 价值

- 旁路观测、录制、镜像流失败，不影响用户主 SSE

这就是“主链路优先”的保底思想。

---

### 6.2 SSE 连接失败保底

- 关键类：
  - `src/main/java/com/example/interview/stream/InterviewSseEmitterSender.java`

#### 已有保底行为

1. 发送失败时静默关闭 emitter
2. `complete / completeWithError` 都做了状态保护
3. 对容器生命周期竞争引发的 `IllegalStateException` 直接忽略

#### 价值

- 前端断开、容器异步状态 race 不会把服务线程搞崩

---

### 6.3 流式任务取消保底

- 关键类：
  - `src/main/java/com/example/interview/service/ChatStreamingService.java`
  - `src/main/java/com/example/interview/stream/InterviewStreamTaskManager.java`

#### 已有保底行为

1. 用户可以 stop 当前任务
2. stop 后会：
   - 更新占位消息为“已停止生成”
   - 标记 generationStatus = `CANCELLED`
3. 流程各关键节点都会检查 `isCancelled`
4. 取消后不会继续输出后续 token

#### 价值

- 用户主动打断时，链路能正常收尾，不会继续扣模型费用或刷屏

---

### 6.4 流式失败收口保底

## 13. 当前项目保底策略总结

如果把全项目现有保底策略做一句话总结，可以概括为：

> 能提前拦截的就在入口层拦截；能局部失败的就不要扩大成全链路失败；能重试的先重试；能换候选的换候选；能降级的降级成稳定结构；降级后仍要保证用户可感知、前端可消费、运维可追踪。

从成熟度来看，当前做得比较强的模块是：

1. 模型路由层
   - 候选切换、首包探测、熔断、fallback model 都比较完整
2. MCP 层
   - 超时、重试、stub、审计、统一错误语义都比较成熟
3. RAG 层
   - 检索并行隔离、Web fallback、评估 JSON 降级都比较实用
4. 流式输出层
   - 取消、失败收口、主通道优先做得比较稳

相对还可以加强的地方有：

1. IM 适配器缺少未命中平台的显式日志
2. 会话仓库的 `Optional.empty()` 语义还不够细
3. 限流还是单机内存级
4. 一些 fallback 还是“返回空列表/空结果”，可以进一步做更强提示

---

## 14. 建议你在面试里怎么讲

可以直接这样表达：

> 我们项目的异常保底不是简单 try-catch，而是分层做的。入口层有统一异常处理和限流，意图层有规则前置、意图树回退和澄清机制，模型层有多候选切换、首包超时探测、三态熔断和 fallback 模型，RAG 层有检索通道隔离、Web fallback 和结构化 JSON 降级，MCP 层有参数归一化失败兜底、超时重试、stub 降级和审计记录，流式链路还有取消、失败收口和主通道优先。整体目标是保证用户有反馈、前端有稳定结构、运维有排查依据。

---

## 15. 结论

当前项目已经具备比较完整的“线上可用型保底体系”，不是单点兜底，而是：

- 有前置防御
- 有中间层降级
- 有尾部统一收口
- 有审计和 trace 兜底

如果后续继续增强，我建议优先做三件事：

1. 把限流从单机内存升级到 Redis 分布式限流
2. 把“静默失败”点补成“有日志、有审计、有状态”
3. 对高风险场景再细分 fallback 策略，比如“知识问答可以 Web fallback，面试评分尽量不要外网兜底”

这样这套保底体系就会更接近真正的大厂线上标准。
