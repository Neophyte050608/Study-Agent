# 项目涉及设计模式

本文基于当前项目代码梳理实际使用到的设计模式，重点不是背书式定义，而是结合你项目里的真实类、调用链和职责分工来总结。整体看，这个项目的服务层和运行时编排做得比较像“大厂后端常见的工程化模式组合”：用策略模式做可替换决策，用模板方法固定主流程，用外观模式收敛复杂子系统，用展示器模式隔离“业务结果”和“展示文本”，再配合观察者、适配器、注册表、责任链、工厂、组合、状态机等模式完成扩展性建设。

## 1. 策略模式

### 1.1 模式定义

策略模式的核心是把“可变的算法/选择逻辑”从主流程里抽出来，封装成可替换的策略。调用方不需要知道具体细节，只需要在合适时机选择策略并执行。

### 1.2 项目中的落地

#### 1.2.1 MCP 网关选择策略

- 关键类：
  - `src/main/java/com/example/interview/service/mcp/GatewaySelectionStrategy.java`
  - `src/main/java/com/example/interview/service/McpGatewayService.java`

`GatewaySelectionStrategy` 专门负责根据 `mode`、能力名、当前可用网关，决定走哪个 `McpCapabilityGateway`。  
例如会在 `stub / bridge / sse / stdio / fastmcp / auto` 之间切换；如果是数据库类 capability，还会优先转发给 `DatabaseMcpAdapterRouter`。

这就是很典型的策略思想：

- 把“选择哪种网关”的变化点从 `McpGatewayService` 主流程中拆出去
- 主服务只关心“拿到一个网关后怎么调”
- 新增网关类型时，扩展点集中在策略类，而不是把 `if-else` 散落到整个服务层

可以把这里总结为：

> 项目把 MCP 调用中的通道选择逻辑抽成了独立策略，业务层只依赖统一网关接口，避免服务层和具体传输协议强耦合。

#### 1.2.2 模型 Bean 解析策略

- 关键类：
  - `src/main/java/com/example/interview/service/model/ModelBeanResolver.java`
  - `src/main/java/com/example/interview/service/DynamicModelFactory.java`

`ModelBeanResolver` 根据 provider、beanName 决定最终使用哪个 Spring Bean，例如 `openAiChatModel`、`zhiPuAiChatModel`、`ollamaChatModel`。

虽然这个实现没有拆成多个策略类，但职责上已经是“策略封装”：

- 同样输入，不同 provider 走不同解析规则
- `DynamicModelFactory` 不直接判断 provider
- 解析规则集中收敛在 `ModelBeanResolver`

这在面试里可以说成“轻量级策略模式”或“策略对象化的解析器”。

### 1.3 使用收益

- 降低了服务层中的条件分支复杂度
- 新增接入通道或模型提供商时修改面更集中
- 有利于做灰度、回退、自动选择

### 1.4 注意点

当前 `GatewaySelectionStrategy` / `ModelBeanResolver` 还是偏 `if-else` 型策略，如果后续通道和 provider 更多，可以进一步演进成：

- `Map<String, GatewayStrategy>`
- `Map<String, ModelResolveStrategy>`

这样开放封闭性会更强。

## 2. 模板方法模式

### 2.1 模式定义

模板方法模式的核心是：先定义一套稳定的流程骨架，把其中可变步骤交给不同实现去完成，或者以参数/回调形式注入。

### 2.2 项目中的落地

#### 2.2.1 路由执行模板

- 关键类：
  - `src/main/java/com/example/interview/modelrouting/RoutingExecutionTemplate.java`

`RoutingExecutionTemplate#execute` 固定了这样一套流程：

1. 判断路由是否启用
2. 选择候选模型
3. 候选为空则走 fallback
4. 否则交给执行器按候选顺序调用

这里的变化点通过两个函数参数注入：

- `fallbackSupplier`
- `candidateExecutor`

这就是非常典型的模板方法风格：  
流程骨架稳定，可变步骤由调用方传入。

#### 2.2.2 入库任务流水线模板

- 关键类：
  - `src/main/java/com/example/interview/ingestion/IngestionTaskService.java`
  - `src/main/java/com/example/interview/ingestion/IngestionStageNode.java`
  - `src/main/java/com/example/interview/ingestion/IngestionStageNodeRegistry.java`

`IngestionTaskService#runPipeline` 固定了入库任务的主流程：

1. 创建执行上下文
2. 按 pipeline stages 顺序遍历
3. 根据 stage 从注册表获取节点
4. 执行节点
5. 记录阶段日志
6. 异常则记录失败节点
7. 最后校验 summary

这里每个阶段真正做什么，交给 `IngestionStageNode` 的不同实现：

- `FetchStageNode`
- `ParseStageNode`
- `EnhanceStageNode`
- `ChunkStageNode`
- `PassThroughStageNode`
- `SyncCommitStageNode`

这也是模板方法思想，只不过模板骨架在服务层，具体步骤通过“阶段节点对象”实现。

### 2.3 使用收益

- 主流程清晰稳定
- 节点逻辑可替换、可扩展
- 非常适合“多阶段任务编排”“多候选重试”“统一观测埋点”

### 2.4 面试表达

> 项目里我没有把所有流程硬写在一个大方法里，而是抽出执行模板。比如模型路由和知识入库都先定义流程骨架，再把变化点封装成可插拔步骤，这样后续接新能力时不会破坏主链路。

## 3. 外观模式

### 3.1 模式定义

外观模式的核心是对复杂子系统提供一个更简洁、稳定的统一入口，降低调用方理解和使用成本。

### 3.2 项目中的落地

#### 3.2.1 MCP 服务外观

- 关键类：
  - `src/main/java/com/example/interview/service/McpGatewayService.java`

`McpGatewayService` 对外只暴露：

- `discoverCapabilities`
- `invoke`

但内部实际屏蔽了很多复杂度：

- 网关选择
- 参数归一化
- 超时执行
- 重试
- fallback to stub
- 审计记录

调用方不需要知道底层到底是 `SSE`、`stdio`、`bridge`、`fastmcp` 还是数据库适配器，只需要调这个服务。

这就是外观模式非常标准的使用方式。

#### 3.2.2 Application Service 外观层

- 关键类：
  - `src/main/java/com/example/interview/service/interview/McpApplicationService.java`
  - `src/main/java/com/example/interview/service/interview/InterviewSessionApplicationService.java`
  - `src/main/java/com/example/interview/service/interview/ObservabilityApplicationService.java`
  - `src/main/java/com/example/interview/service/interview/ProfileApplicationService.java`

这些类本质上都是“应用层外观”：

- 对 controller 提供面向用例的接口
- 屏蔽 agent、service、路由、开关判断、DTO 装配细节
- 让上层调用更稳定

其中 `InterviewSessionApplicationService` 很典型，它把：

- 任务构造
- `TaskRouterAgent` 调度
- 结果判空/失败处理
- 视图对象转换

统一收敛起来。controller 不需要知道后面到底经过了多少 agent/service。

### 3.3 使用收益

- 上层接口更稳定
- 子系统复杂度被收口
- 有利于控制器瘦身和边界清晰

## 4. 展示器模式

### 4.1 模式定义

展示器模式的核心思想是把“业务结果对象”与“最终呈现文本/视图格式”解耦。  
也就是业务层负责产生语义化结果，展示器负责转成面向渠道的展示内容。

### 4.2 项目中的落地

#### 4.2.1 任务响应展示器

- 关键类：
  - `src/main/java/com/example/interview/service/TaskResponsePresentationService.java`

`TaskResponsePresentationService` 会把统一的 `TaskResponse` 转成最终展示文本，并按不同渠道区分格式：

- `WEB`
- `IM`

它识别多种业务结果类型：

- `InterviewSession`
- `InterviewOrchestratorAgent.AnswerResult`
- `InterviewOrchestratorAgent.FinalReport`
- `Map` 型通用结果

这意味着业务层可以先返回结构化结果，展示层再决定文案和输出格式，而不是让每个 Agent/Service 都自己拼字符串。

#### 4.2.2 IM 展示器

- 关键类：
  - `src/main/java/com/example/interview/im/runtime/ImResponsePresenter.java`

`ImResponsePresenter` 进一步把 IM 渠道格式化单独封装出来，本质上是针对 IM 场景的 presenter。

### 4.3 为什么这里很像 MVP 中的 Presenter

你这个项目不完全是传统前端 MVP，但思想是一致的：

- Agent / Service 负责业务
- Presenter 负责展示组织
- Controller / Webhook 只负责接线

这是非常工程化的分层方式。

### 4.4 面试表达

> 我把任务执行结果和最终展示文本分开了，统一由 `TaskResponsePresentationService` 做格式编排。这样同一个业务结果可以适配 Web、IM 等多个渠道，避免业务代码里到处拼接文案。

## 5. 观察者模式

### 5.1 模式定义

观察者模式是一对多通知机制。主题对象状态变化后，会把事件广播给多个观察者。

### 5.2 项目中的落地

- 关键类：
  - `src/main/java/com/example/interview/stream/StreamEventEmitter.java`
  - `src/main/java/com/example/interview/stream/ObservableStreamEmitter.java`
  - `src/main/java/com/example/interview/stream/InterviewSseEmitterSender.java`
  - `src/main/java/com/example/interview/service/ChatStreamingService.java`

`ObservableStreamEmitter` 内部维护 `observers` 列表，事件发生时：

- 先通知主发送器 `primary`
- 再逐个通知观察者 `observer`

并且观察者失败不会影响主流输出，这种“主通道 + 旁路观察者”的实现很适合：

- SSE 实时输出
- 埋点/日志
- 调试镜像流
- 后续录制回放

你提到的“SSE 流式输出应该用到了观察者模式”，判断是成立的，而且落点很明确。

### 5.3 这里还有一点装饰器味道

`ObservableStreamEmitter` 本身也包裹了一个 `primary` emitter，并对原 emitter 增强了“广播给旁路观察者”的能力，因此它既体现了观察者模式，也带有装饰器模式的结构特征。

## 6. 责任链模式

### 6.1 模式定义

责任链模式把多个处理器串成链，请求沿链传递，直到有一个处理器接住并处理完成。

### 6.2 项目中的落地

- 关键类：
  - `src/main/java/com/example/interview/service/chatstream/ChatScenarioHandler.java`
  - `src/main/java/com/example/interview/service/chatstream/ChatScenarioHandlerRegistry.java`
  - `src/main/java/com/example/interview/service/chatstream/handler/ActiveInterviewChatScenarioHandler.java`
  - `src/main/java/com/example/interview/service/chatstream/handler/ActiveCodingChatScenarioHandler.java`
  - `src/main/java/com/example/interview/service/chatstream/handler/KnowledgeQaChatScenarioHandler.java`
  - `src/main/java/com/example/interview/service/chatstream/handler/DefaultRoutedChatScenarioHandler.java`

`ChatScenarioHandlerRegistry#handle` 会按照顺序遍历 handler，谁返回 `true` 就终止链路。

实际链路大致是：

1. 先判断是不是已有进行中的面试会话
2. 再判断是不是已有进行中的编码会话
3. 再判断是不是知识问答流式委派
4. 最后走默认路由处理

这就是非常标准的责任链模式。  
它特别适合“同一请求可能命中多种场景”的流式对话入口。

### 6.3 同类思想在任务分发里也存在

- 关键类：
  - `src/main/java/com/example/interview/agent/router/TaskHandler.java`
  - `src/main/java/com/example/interview/agent/router/TaskHandlerRegistry.java`
  - `src/main/java/com/example/interview/agent/TaskRouterAgent.java`

这里不是严格的链式遍历，而是“先注册、再按类型路由到对应 handler”，更像命令分发/注册表模式，但和责任链一样，都在解决“如何把请求交给正确处理者”的问题。

## 7. 适配器模式

### 7.1 模式定义

适配器模式用于把不兼容接口包装成统一接口，让调用方按一致方式使用不同实现。

### 7.2 项目中的落地

#### 7.2.1 IM 平台适配器

- 关键类：
  - `src/main/java/com/example/interview/im/runtime/ImPlatformAdapter.java`
  - `src/main/java/com/example/interview/im/runtime/FeishuPlatformAdapter.java`
  - `src/main/java/com/example/interview/im/runtime/QqPlatformAdapter.java`
  - `src/main/java/com/example/interview/im/runtime/ImPlatformAdapterRegistry.java`

飞书和 QQ 的发送能力不同，但系统统一抽成 `ImPlatformAdapter#sendReply`。  
这样上层只需要面向统一接口发送 `UnifiedReply`。

#### 7.2.2 数据库 MCP 适配器

- 关键类：
  - `src/main/java/com/example/interview/tool/DatabaseMcpAdapter.java`
  - `src/main/java/com/example/interview/tool/DatabaseMcpAdapterRouter.java`

不同数据库能力按 namespace + capability 适配为统一的 `DatabaseMcpAdapter` 协议，再由路由器统一转发。

### 7.3 使用收益

- 屏蔽第三方平台差异
- 降低业务对外部 SDK 的侵入
- 平台替换成本更低

## 8. 工厂模式

### 8.1 模式定义

工厂模式负责对象创建，把“如何实例化复杂对象”从使用方剥离出去。

### 8.2 项目中的落地

- 关键类：
  - `src/main/java/com/example/interview/service/DynamicModelFactory.java`
  - `src/main/java/com/example/interview/service/model/DynamicChatModelRegistry.java`

`DynamicModelFactory` 负责根据 Agent 配置或路由候选信息返回合适的 `ChatModel`。  
当候选带有 `baseUrl/apiKey/model` 时，还会动态创建模型实例；创建细节进一步由 `DynamicChatModelRegistry` 封装和缓存。

这是典型的工厂职责：

- 创建逻辑复杂
- 调用方不关心实例化细节
- 可以结合缓存减少重复构建

## 9. 注册表模式

### 9.1 模式定义

注册表模式通过一个中心容器管理“名称/类型 -> 实现”的映射，解决运行时查找和扩展问题。

### 9.2 项目中的落地

项目里注册表模式非常多，说明整体架构是偏插件化、可装配的：

- `TaskHandlerRegistry`
  - `TaskType -> TaskHandler`
- `ChatScenarioHandlerRegistry`
  - 有序 `ChatScenarioHandler` 列表
- `IngestionStageNodeRegistry`
  - `IngestionNodeStage -> IngestionStageNode`
- `IngestionPipelineRegistry`
  - `sourceType -> IngestionPipelineDefinition`
- `ImPlatformAdapterRegistry`
  - `platform -> ImPlatformAdapter`
- `DynamicChatModelRegistry`
  - `candidateName -> ChatModel`

这类模式的价值很大：

- 新增实现时基本只用注册，不用改主流程
- 运行时可按 key 查找正确处理器
- 很适合做多渠道、多任务类型、多阶段流程

## 10. 组合模式

### 10.1 模式定义

组合模式用于把多个同类对象组合成一个更大的统一对象，对外表现成同一接口。

### 10.2 项目中的落地

- 关键类：
  - `src/main/java/com/example/interview/modelrouting/CandidateProvider.java`
  - `src/main/java/com/example/interview/modelrouting/CompositeCandidateProvider.java`

`CompositeCandidateProvider` 同时聚合：

- `YamlCandidateProvider`
- `DatabaseCandidateProvider`

对外仍然只暴露统一的 `CandidateProvider#getCandidates`。

这就是组合模式：

- 子节点是多个候选提供者
- 组合对象把它们合并成一个统一提供者
- 上层 `ModelSelector` 无需关心候选来自 YAML 还是数据库

## 11. 状态机模式

### 11.1 模式定义

状态机模式把对象状态、事件、状态转移规则显式化，避免用大量 if-else 管理复杂流程。

### 11.2 项目中的落地

- 关键类：
  - `src/main/java/com/example/interview/core/statemachine/InterviewStateMachineConfig.java`

项目使用 COLA StateMachine 显式描述面试 SOP：

- `INTRODUCTION -> RESUME_DEEP_DIVE`
- `RESUME_DEEP_DIVE -> CORE_KNOWLEDGE`
- `CORE_KNOWLEDGE -> SCENARIO_OR_CODING`
- `SCENARIO_OR_CODING -> CLOSING`

并把：

- 状态
- 事件
- 条件
- 动作

拆开管理。  
这比把面试阶段流转散落在业务代码里更稳定，也更容易调试和扩展。

## 12. 代理/切面模式

### 12.1 模式定义

代理模式强调“控制对目标对象的访问”，AOP 切面则常用于在不侵入业务代码的前提下织入横切逻辑，如日志、事务、链路追踪。

### 12.2 项目中的落地

- 关键类：
  - `src/main/java/com/example/interview/observability/TraceNode.java`
  - `src/main/java/com/example/interview/observability/TraceNodeAspect.java`

`@TraceNode` + `TraceNodeAspect` 的组合，本质上是通过 AOP 代理为方法织入 trace 节点记录逻辑：

- 方法执行前创建 trace node
- 成功时记录 success
- 异常时记录 fail

这是很典型的横切关注点抽离。  
从设计模式角度，可以归为“代理/AOP 装饰式增强”。

## 13. 命令分发思想

虽然项目里没有出现经典的 `Command` 命名，但下面这组结构有明显命令模式味道：

- `TaskRequest`
- `TaskResponse`
- `TaskHandler`
- `TaskRouterAgent`

每个 `TaskHandler` 对应一种 `TaskType`，`TaskRouterAgent` 负责找到处理者并执行。  
这本质上是在把“请求”对象化，再把执行逻辑交给对应命令处理器。

在面试中可以表达为：

> 项目采用了命令式任务分发模型，把任务类型、载荷、上下文统一抽象成 `TaskRequest`，再通过注册表定位对应处理器，避免入口层直接依赖具体业务实现。

## 14. 这份项目最值得重点讲的设计模式组合

如果面试官让你讲“项目里最有代表性的设计模式”，我建议重点讲下面这组，不要平均用力：

1. 策略模式
   MCP 网关选择、模型 Bean 解析，解决多通道/多 provider 的切换问题。
2. 模板方法模式
   路由执行模板、入库流水线模板，解决稳定流程与可变步骤解耦问题。
3. 外观模式
   `McpGatewayService` 和多个 `ApplicationService` 收口复杂子系统。
4. 展示器模式
   `TaskResponsePresentationService` / `ImResponsePresenter` 负责结果展示编排。
5. 观察者模式
   `ObservableStreamEmitter` 支撑 SSE 事件广播。
6. 责任链模式
   `ChatScenarioHandlerRegistry` 实现多场景流式处理分发。

这六个模式串起来，基本就能把你这次“服务层重构”的设计思路讲清楚。

## 15. 一段可直接复述的项目总结

可以直接这样说：

> 这个项目在服务层重构里，核心是把复杂流程拆成稳定骨架和可替换组件。  
> 我们用策略模式解决 MCP 网关和模型解析的动态选择，用模板方法模式固化模型路由和知识入库的主流程，用外观模式把复杂子系统收敛成应用服务入口，用展示器模式把业务结果与多渠道展示解耦。  
> 在流式输出场景里，SSE 事件发送采用了观察者模式；在聊天场景识别上，采用责任链模式依次匹配不同对话处理器；在平台接入上，则通过适配器模式统一飞书、QQ 和数据库类 MCP 能力。整个架构的特点是高扩展、低耦合、便于灰度和演进。

## 16. 结论

这个项目不是“为了用设计模式而用模式”，而是在以下真实工程问题下自然演化出了这些模式：

- 多模型、多网关、多平台接入
- 多场景对话分发
- 流式输出与异步观测
- 应用层接口稳定化
- 入库流程任务化、阶段化、可观测化

因此项目里最重要的不是死记模式名称，而是要讲清楚：

- 你解决了什么变化点
- 为什么要把变化点抽离
- 抽离后扩展成本如何下降
- 主流程如何保持稳定

这才是大厂面试里真正加分的表达方式。
