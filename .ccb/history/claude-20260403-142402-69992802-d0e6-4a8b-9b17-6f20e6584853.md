### 单用户后可考虑的优化

**1. 文件式记忆 + autoDream 整理**
- **原拒绝理由**：多用户并发写文件会冲突，不兼容容器化部署
- **单用户后**：完全可行。可以像 Claude Code 一样用 `MEMORY.md` + YAML frontmatter 存储用户画像，用 autoDream 定期整理合并记忆（后台 LLM 调用把零散记忆整合为结构化档案）
- **与现有 P2-A 的关系**：P2-A 已用 DB 实现了跨会话记忆，但 autoDream 的**自动整理/遗忘**机制是 P2-A 缺少的——当前记忆只会不断合并变大，没有"过期淘汰"和"重要性排序"

**2. 安全管线简化**
- **原设计理由**：多用户需要 JWT 认证 + 每用户速率限制 + 每 IP 限流
- **单用户后**：JWT 认证和速率限制都可以去掉（自己不会攻击自己）。只保留 InputSanitizer（prompt injection 防护仍有价值，因为 LLM 本身不区分用户数）
- **具体可删**：`RateLimitInterceptor`、`SecurityConfig` 的 JWT 模式、`UserIdentityResolver` 的 JWT 解析链路

**3. Prompt Cache 优化（SYSTEM_PROMPT_DYNAMIC_BOUNDARY）**
- **原拒绝理由**：Spring AI 不暴露 OpenAI provider 的 cache control A​PI
- **单用户后**：如果切换为直接调用 OpenAI/DeepSeek A​PI（绕过 Spring AI），就可以利用 A​PI 级别的 prompt caching——将系统提示词标记为 static 部分全局缓存，每次只传 dynamic 部分。**单用户下缓存命中率极高**（只有一个人在用），token 成本可以降 50%+
- **前提**：需要绕过 Spring AI 直接用 HTTP client 调 A​PI，改造量较大

---

### 单用户后仍然不适用的

| 模式 | 原因 |
|------|------|
| **YoloClassifier** | Agent 不执行任意工具/Shell，与用户数无关 |
| **Fork Agent 共享 prompt cache** | Node.js 进程级优化，Java 无对等机制 |
| **工具权限白名单 + 沙箱** | Agent 间是内部方法调用，非动态工具发现 |

---

**总结**：单用户真正解锁的是 **autoDream 记忆整理**和 **A​PI 级 prompt cache**。前者成本低（1-2 天），后者收益高但改造大。安全管线简化则是减负——可以删掉不需要的代码。
