## 方案对比

### 方案 A：Ragas 指标级中文 Prompt 覆写（推荐）

Ragas 0.2.x 每个 metric 对象支持自定义 prompt。在 `app.py` 中评测前覆写 prompt 为中文版本：

```python
faithfulness.faithfulness_statements_prompt = "中文版 claim 拆分 prompt"
faithfulness.nli_statements_prompt = "中文版 NLI 判定 prompt"
```

- **难度**：★★★☆☆（需要阅读 Ragas 源码提取原版 prompt，逐一翻译适配）
- **效果**：★★★★★（直接解决核心问题，LLM 用中文理解中文内容）
- **风险**：Ragas 版本升级时 prompt 字段名可能变化
- **工作量**：修改 `app.py`，约 100-150 行 prompt 模板

### 方案 B：Ragas `adapt` + `language` 参数

Ragas 部分版本支持 `metric.adapt(language="chinese", llm=llm)` 自动翻译内部 prompt。

- **难度**：★☆☆☆☆（一行代码）
- **效果**：★★★☆☆（自动翻译质量不可控，可能不如手工适配）
- **风险**：不是所有 Ragas 版本都支持；自动翻译的 prompt 可能有偏差
- **工作量**：修改 `app.py`，约 10 行

### 方案 C：换用更强的 LLM 做评测

Ragas 评测用的 LLM 不一定要和业务 LLM 相同。可以给 Ragas 配一个对结构化英文 prompt 遵循度更高的模型（如 GPT-4o、Claude）。

- **难度**：★☆☆☆☆（改配置即可）
- **效果**：★★★★☆（GPT-4 对 Ragas 原版 prompt 的遵循度远高于 DeepSeek）
- **风险**：增加 API 成本；需要额外的 API Key
- **工作量**：改 `docker-compose.yml` 环境变量

### 方案 D：双引擎加权融合评分

接受两个引擎的评分差异，在前端或后端引入加权融合逻辑。例如 Java 引擎权重 0.6 + Ragas 权重 0.4，或取两者最大值。

- **难度**：★★☆☆☆
- **效果**：★★☆☆☆（治标不治本，不解决 Ragas 分数偏低的本质问题）
- **工作量**：修改前后端各少量代码

### 方案 E：A + B 组合（最佳实践）

先尝试 `adapt(language="chinese")`（方案 B），如果效果不够，再手工覆写关键 prompt（方案 A）。同时在请求中支持可选的 `language` 参数，让用户可以选择中文/英文模式。

- **难度**：★★★☆☆
- **效果**：★★★★★
- **工作量**：修改 `app.py` + `EvalRequest` 模型
