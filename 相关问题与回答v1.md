## 详细解答（结合当前项目实现）

### 1. AI Agent 核心架构与设计

1. 关于长期记忆、短期记忆、元记忆的划分  
当前项目没有把“记忆”混成一个桶，而是按使用时效和可验证性分层：
   - 短期记忆进入会话上下文，主要落在 `t_chat_message` 与 `t_chat_session`，由 `ChatContextCompressor` 做压缩。适合放最近几轮对话、当前任务约束、澄清状态、当前 session 的局部目标。
   - 长期记忆进入结构化存储或检索存储，而不是盲目全量进向量库。比如用户稳定偏好、能力特征进入 `t_user_chat_memory`；学习事件进入 `t_learning_event`/`t_learning_profile`；知识文档进入父子索引和向量库。
   - 元记忆我不理解为“在线改模型权重”。当前项目没有运行期直接改权重，而是通过 Prompt、规则阈值、Few-shot 样例、评测参数模板、意图树配置去沉淀系统经验。也就是说，这个项目的元记忆是“系统配置层的学习”，不是在线微调。
   - 一个很实用的判断标准是：是否需要可检索、可追责、可删除。如果需要，优先落存储；只在当前任务有效，则进上下文；如果是跨任务稳定策略，进入模板/规则/配置，不进用户记忆。

2. Agent 是怎么实现的，如何防逻辑塌缩和无效调用  
当前项目的 Agent 不是纯 ReAct 放飞，而是“规则预过滤 + 意图树 + 槽位精炼 + Handler 执行 + A2A 事件观测”的受控编排。
   - 入口是 `TaskRouterAgent`。先跑 `IntentPreFilter`，处理 `/help`、清空上下文、显式开始面试、刷题等高确定性意图。
   - 再走 `IntentTreeRoutingService`，先做域分类，再做域内叶子分类，而不是让模型在全量任务上裸分类。
   - 分类后再跑槽位精炼，把 `topic`、`difficulty`、`count`、`questionType` 这类参数补齐。
   - 真正执行不是让模型自由调用任意工具，而是落到 `TaskHandlerRegistry` 中受控 Handler。
   - 为避免逻辑塌缩，我做了几层保护：
     - 高确定性场景优先规则，不让模型在简单问题上“过度思考”。
     - 低置信场景主动澄清，不硬判。
     - 工具/任务执行有明确 receiver，不让模型无限生成“下一步我要继续调用工具”。
     - 模型调用失败或分类失败时回退 legacy 路由，而不是直接报废主流程。
     - MCP 调用前做参数标准化，调用后做失败分类、重试、stub fallback。
   - 本质上，纠错不是靠模型“自省一句话”，而是靠工程上把自由度收窄，把错误面前移。

3. 什么是 Agent，它和直接调 API 的区别  
直接调 API，本质是“给模型一个 prompt，拿回一个回答”。  
Agent 是“带状态、带目标、带工具、带观测、带恢复策略的任务执行体”。  
当前项目里 Agent 至少多了五层能力：
   - 有任务目标，不只是文本续写。
   - 有状态机，例如面试中、澄清中、流式执行中。
   - 有工具和知识访问能力。
   - 有失败处理和降级路径。
   - 有记忆和跨轮承接。
所以我对外会说：API 是推理接口，Agent 是围绕推理接口构建的任务系统。

4. 了解的 Agent 架构模式  
常见有几类：
   - ReAct：思考-行动-观察循环。
   - Plan-and-Execute：先规划，再分步执行。
   - Router + Specialist：先路由到专业子代理。
   - Workflow Agent：固定编排，模型只做局部决策。
   - Multi-Agent：多个代理协同，通过消息总线通信。
   - Tool-Augmented RAG Agent：先检索证据，再规划回答/动作。
当前项目实际采用的是 Router + Specialist + Workflow 的混合模式，而不是完全开放式 ReAct。

5. Agent 的任务规划怎么做  
当前项目规划分三层：
   - 规则层：明确命令直接短路。
   - 轻规划层：意图树决定“要做哪类任务”。
   - 业务层：具体 Handler 内部决定执行步骤，例如面试是“起题 -> 作答 -> 评估 -> 追问/下一题 -> 报告”。
也就是说，规划不是一次性生成整棵计划树，而是按业务阶段做局部规划。

6. 规划是模型做还是规则做  
不是二选一，而是混合：
   - 确定性高的，由规则做。
   - 语义模糊但边界清晰的，由小模型/轻模型做分类和槽位抽取。
   - 生成性强、需要知识和推理的部分，由大模型做。
工程上最差的做法是把所有决策都交给模型。当前项目刻意避免这一点。

7. 任务怎么拆，粒度怎么定  
我的原则是按“可观测、可回退、可复用”拆，而不是按类数量拆。
   - 面试编排拆成 `InterviewOrchestratorAgent`、评估层、知识层、成长层，是因为它们天然阶段不同、失败模式不同。
   - 路由拆成 pre-filter、intent-tree、slot refine，是因为它们的误差类型不同。
   - 检索拆成 rewrite、retrieval、generation，是为了观测和评测。
粒度判断标准有三个：
   - 这一段是否可以独立失败和降级。
   - 这一段是否需要独立指标。
   - 这一段未来是否会替换实现。

8. 系统中的 Agent 状态  
如果从业务上看，至少包括：
   - 待路由
   - 已识别意图
   - 待澄清
   - 执行中
   - 流式输出中
   - 已完成
   - 失败
   - 已取消
如果从 A2A 消息状态看，项目里已有 `PENDING / PROCESSING / DONE / FAILED`。  
如果从模型熔断看，还有 `CLOSED / OPEN / HALF_OPEN`。  
如果从面试会话看，还有开始、答题中、结束、报告生成中等状态。

9. 是否支持多租户同时调用  
当前代码层面支持多用户并发调用，核心隔离键是 `userId + sessionId`，数据存储也按用户或会话分离。  
但严格意义上的 SaaS 多租户还没有完全做完，比如：
   - 模型配置还不是强租户命名空间隔离。
   - MCP capability 不是按租户动态授权。
   - 向量库/知识库不是天然多租户 schema。
所以我会诚实表述：现在是“多用户并发 + 基础隔离”，不是“完整企业级多租户治理”。

10. 为什么设计多个 Agent，各自职责  
这是因为不同任务的约束完全不同：
   - `TaskRouterAgent` 负责理解用户到底想干什么。
   - `InterviewOrchestratorAgent` 负责面试流程总控。
   - `KnowledgeQaAgent` 负责知识问答。
   - `CodingPracticeAgent` 负责刷题与代码题流程。
   - `LearningProfileAgent` 负责画像沉淀和学习计划。
   - `ChatMemoryExtractorAgent`/`ChatContextCompressor` 负责上下文治理与长期记忆提炼。
如果硬做成单 Agent，问题不是“代码不好看”，而是不同任务会互相污染 prompt、状态和错误恢复逻辑。

11. 当前 Agent 是否达到预期  
我认为达到“可演示、可扩展、可继续工程化”的预期，但还没达到“高度自治”的预期。
   - 现在强项是：链路清晰、业务闭环完整、可观测、可评测、能稳定跑。
   - 当前短板是：工具自治度还不够高，复杂规划更多靠工作流和路由，不是强泛化 Agent。
我对 Agent 的预期不是“像人一样自由思考”，而是“在明确边界内高成功率完成任务，并且失败可定位、可恢复、可优化”。

### 2. RAG 架构与检索流程

1. 为什么引入父子索引  
因为纯扁平 chunk 检索有两个典型问题：
   - chunk 太小，命中虽准但缺语境。
   - chunk 太大，语义被稀释，召回不稳。
当前项目用 `t_rag_parent + t_rag_child`，先召回 child，再回填 parent 上下文。这样能同时保留“命中点”和“章节语境”。这比直接把整段 parent 塞给模型更稳，因为系统还会构造“父文上下文 + 命中片段”的证据，而不是粗暴覆盖。

2. 为什么引入 BM25  
因为向量检索对精确术语、版本号、缩写、函数名、配置名并不总是稳定。  
像 MySQL 索引、Redis 命令、JVM 参数、类名、表名这类词，词法召回往往比纯 embedding 更可靠。  
所以项目里保留 `LexicalIndexService`，并支持 `AUTO / FULLTEXT / LIKE`。

3. BM25 和向量怎么组合，比例怎么设  
当前项目不是固定 0.7/0.3 这种线性加权，而是多路召回后走 RRF 融合，再做去重和 rerank。  
这意味着“比例”不靠人工常数，而靠名次融合。好处是不同通道打分口径不一致时，RRF 更稳，不需要强行统一分值空间。  
当前主链路包含：
   - 意图定向向量通道
   - 全局向量通道
   - 词法检索
   - 图谱检索
   - 必要时 Web fallback

4. 从 query 到最终上下文的完整流程  
当前项目中的完整链路是：
   - 用户 query 进入 `RAGService.buildKnowledgePacket`
   - 先做 query rewrite，拆成 `coreTerms + expandTerms + fullQuery`
   - 并行做混合召回：向量、词法、图谱、父子文档回填、图片关联
   - 对多路结果做 RRF 融合
   - 去重
   - 基于 query overlap、metadata、命中位置做 rerank
   - 计算 best score，决定是否触发 Web fallback
   - 构建 `context / imageContext / retrievalEvidence`
   - 生成阶段严格约束引用 evidence 编号
这个流程的关键不是“检索出来”，而是“把检索结果整理成可消费的证据上下文”。

5. 是否做 rerank，用的什么方式  
做了，但目前主要是工程型 rerank，不是重 cross-encoder。
   - 先 RRF 融合。
   - 再做重复去重。
   - 再根据 query overlap、元数据、父子命中片段做排序。
这是一个成本和效果之间的折中。后续如果要继续提相关度，我会在高价值场景上引入 cross-encoder reranker，但不会默认全链路上，因为时延和成本会上去。

6. RAG 的主要模式和工作流程  
结合这个项目，我会分成五类：
   - 朴素 RAG：检索后直接拼上下文。
   - Hybrid RAG：词法 + 向量 + 图谱混合。
   - Parent-Child RAG：细粒度召回 + 粗粒度回填。
   - Multimodal RAG：文本和图片联合召回。
   - Evaluated RAG：检索链路带评测、留档、趋势分析。
当前项目其实已经从“回答系统”升级成“可评测的 RAG 平台”。

7. Agent 一般在什么阶段查向量库，怎么查  
不是所有 Agent 都查。  
一般在知识问答、回答评估、首题生成、代码解释、图文查询这些阶段查。  
查询方式是服务层调用，不是把底层向量 API 直接暴露给大模型自由拼接。核心入口就是 `RAGService.buildKnowledgePacket(...)`。

8. 查询知识库工具函数的标准输入输出  
以当前项目的服务语义看：
   - 输入：`question`、可选 `userAnswer`、是否允许 `webFallback`
   - 输出：`KnowledgePacket`，里面至少包括 `rewrittenQuery`、`retrievedDocs`、`retrievedImages`、`context`、`imageContext`、`retrievalEvidence`、`webFallbackUsed`
也就是说，我不会只返回“文档列表”，而是返回已经适配生成层的知识包。

9. 怎么做多模态知识检索  
当前项目做的是“图文联合召回”，不是简单上传图片给大模型看。
   - 入库时用 `ImageReferenceExtractor` 从 Markdown 抽图片引用。
   - `VisionModelService`/图片流水线生成图片摘要和向量。
   - 文本块和图片关系落到 `t_text_image_relation`。
   - 查询时既可以“文档命中后反查关联图”，也可以“直接按图像语义检索”。
   - 最终合并到 `imageContext`，供生成阶段引用。
这种设计比单轮多模态问答更适合知识库场景，因为它保留了图和文之间的结构关系。

10. RAG 怎么构建的  
我会概括成四层：
   - 数据层：Markdown/简历/图片等数据入库，构建父子索引、词法索引、图片索引、图谱关系。
   - 检索层：query rewrite、多路召回、RRF、rerank、fallback。
   - 生成层：提示词模板 + 证据编号约束 + 模型路由。
   - 评测观测层：trace、retrieval eval、quality eval、趋势聚类。
真正让系统成熟的不是某个单点算法，而是这四层闭环。

### 3. RAG 优化与性能提升

1. 提升相关度，我会优先动哪几层  
优先顺序通常是：
   - 改 chunk 结构，而不是先换模型。
   - 改 query rewrite 与意图标签过滤。
   - 提高词法召回质量，尤其是术语场景。
   - 优化 parent-child 回填方式。
   - 最后再引入更重的 reranker。
原因是前四项的收益通常更稳定、成本更低。

2. 优化回答效果有哪些思路  
回答效果不只看检索，还看生成约束：
   - 限制模型只引用 evidence 编号。
   - 把冲突证据显式列出来。
   - 上下文按“证据目录 + 正文”组织，而不是原始拼接。
   - 对不同任务走不同 prompt 模板。
   - 图文混合时把图片上下文单独分区，避免干扰文本推理。

3. 该优化检索还是生成  
先看问题在哪。  
如果召回错了，生成再强也只能编。  
如果召回对了但回答差，说明问题在生成约束、上下文工程、提示模板。  
当前项目已经把这两层拆开评测，这就是为了避免“整体效果差，但不知道差在哪”。

4. 如何验证优化有效  
当前项目已有比较完整的方法：
   - 检索层看 Recall@k、MRR、失败聚类、趋势。
   - 生成层看质量评测、证据引用正确率、人工 spot check。
   - 线上看 trace、时延、fallback 率、用户可接受度。
没有评测只靠主观体验，是最容易把系统越改越坏的。

5. RAG 性能如何提升  
三个方向：
   - 检索并行：向量、词法、图谱并发跑。
   - 路由优化：不需要检索的场景别进重链路。
   - 缓存和裁剪：query rewrite cache、上下文截断、只留高价值块。
`RAGService` 已经有 rewrite cache 和并行检索线程池，这是当前项目的已有优化点。

6. 检索、模型、系统架构该怎么分工优化  
   - 检索层负责“拿到对的证据”。
   - 模型层负责“用对证据回答”。
   - 系统层负责“低延迟、稳定、可观测、可降级”。
很多团队把所有问题都丢给更大的模型，这通常是错误的。

7. 有没有缓存或并行优化  
有：
   - query rewrite 有 TTL 缓存。
   - 混合检索走并行执行器。
   - 模型路由支持首包探测和失败自动降级。
   - SSE 让前端感知时间提前，不必等整个任务结束。

8. chunk size 和 top_k 实验怎么做  
我的实验方法不是随便拍脑袋改参数，而是：
   - 固定评测集。
   - 每次只改一组参数，例如 `chunkSize / overlap / topK / lexicalMode / rerank 策略`。
   - 跑 `RetrievalEvaluationService` 留档。
   - 看整体指标和失败聚类，而不是只看平均值。
   - 最终选择不是“平均最高”，而是综合 Recall、稳定性、时延、token 成本。
当前项目已经内置实验模板和趋势对比，这正是为了支撑这种实验方式。

### 4. RAG 评测与数据集

1. RAG 系统怎么评测  
当前项目分两类：
   - 检索评测：验证能不能召回对的证据。
   - 生成质量评测：验证最终回答是否忠实、有用、完整。

2. 评测维度  
至少看：
   - 召回正确性
   - 排序质量
   - 上下文利用率
   - 回答忠实度
   - 回答完整性
   - 时延
   - fallback 触发率

3. 常见指标  
   - Recall@k
   - MRR
   - Hit Rate
   - Faithfulness
   - Answer Relevancy
   - Context Precision / Context Recall
   - 端到端延迟

4. 评测数据集一般包括什么  
一条高质量评测样本至少包括：
   - 用户问题
   - 标准答案或参考答案
   - 期望命中文档/片段
   - 标签，例如主题、难度、来源
   - 可选实验备注与参数快照

5. 如何构建高质量评测数据  
我的原则是：从真实问题里抽，不从理论题库里抄。
   - 真实项目文档
   - 常见面试问法
   - 历史失败 case
   - 模糊问法和别名问法
   - 长尾专业术语
这样测出来的结果才接近真实线上表现。

6. 测试集来源  
当前项目更适合从三类来源构建：
   - 项目知识库本身
   - 常见面试题与技术问答
   - 线上失败样本回灌

7. 测试集量级  
从仓库文档描述看，默认正式评测集已经是 50+ 条，不再是 3 条演示样本。  
我认为这个量级适合做日常回归，但如果要做更稳的参数定版，至少要扩到几百条并覆盖不同主题分布。

### 5. Agent 记忆机制

1. 短期记忆和长期记忆如何区分和存储  
当前项目的短期记忆是会话态，长期记忆是用户态/知识态。
   - 短期：聊天消息、会话摘要、活跃任务状态、澄清上下文。
   - 长期：用户偏好、能力画像、学习事件、知识库文档。

2. 记忆更新策略  
   - 聊天消息实时落库。
   - 会话消息达到阈值后异步压缩摘要。
   - 摘要后再异步提取稳定记忆，写入用户长期记忆。
   - 面试和刷题结果异步沉淀到学习画像。
这是一套“先留原始、后压缩、再提炼”的链路。

3. 长期和短期记忆设计上的区别  
本质区别不是存储介质，而是：
   - 短期记忆服务当前任务，强调上下文连贯。
   - 长期记忆服务跨任务个性化，强调稳定性和可复用。
短期记忆可以丢一些细节，长期记忆必须更保守，因为写错会长期污染系统。

4. 如何记住五轮之前的话  
靠两层：
   - 近期对话原文保留。
   - 更早历史压缩成 `contextSummary`。
如果再往前且属于稳定偏好，就由 `ChatMemoryExtractorAgent` 提炼进长期记忆。  
所以不是无限保留原文，而是“原文 + 摘要 + 结构化记忆”三层协作。

### 6. Agent 工具调用与 MCP

1. Function Calling 怎么设计  
当前项目不是把 function calling 当作一个“万能 JSON 技巧”，而是把它工程化成：
   - 受控 capability 名称
   - 参数标准化
   - 超时
   - 重试
   - 审计
   - fallback
也就是说，函数调用只是表象，关键是后面的治理链路。

2. Function Calling 原理和 Tool Use 的理解  
原理上就是模型输出结构化意图，系统再去执行外部函数，再把结果回灌模型。  
Tool Use 的本质不是“调函数”，而是让模型拥有和外部世界交互的能力。  
关键不在能不能调，而在：
   - 什么时候调
   - 调哪个
   - 参数是否可信
   - 失败怎么收口

3. 参数格式不对怎么办  
当前项目里 `CapabilityParamNormalizer` 会先做标准化和校验。  
如果参数非法，不会把坏请求直接打到下游，而是尽量返回结构化 fallback。  
这是必须的，因为工具错误很多时候不是业务错误，而是“模型生成了非法参数”。

4. 多工具调用顺序怎么定  
顺序由依赖关系决定，而不是由模型即兴发挥。
   - 能并行的并行。
   - 有前置依赖的串行。
   - 高成本工具放后面。
   - 高确定性工具优先。
如果是当前项目，我会先路由/澄清，再检索，再生成，而不会让模型先乱调外部工具再回头理解意图。

5. 工具调用失败如何处理  
当前项目有四层处理：
   - retry
   - retryable / non-retryable 区分
   - fallback to stub
   - 审计记录
最终返回的是结构化失败结果，而不是把异常栈直接暴露给大模型或前端。

6. MCP 与传统 Skills 的区别  
   - Skills 更像“高层能力包”，本质是 prompt、模板、工作流、领域知识。
   - MCP 是标准化工具协议，解决的是模型如何发现、描述、调用外部能力。
简单说，Skill 解决“会不会做”，MCP 解决“能不能安全接外部工具做”。

7. MCP 和框架里 Tool 注册机制区别  
传统 Tool 注册常常是框架内绑定，强耦合语言和运行时。  
MCP 更强调协议化、跨进程、跨语言、动态发现。  
对于这个项目，这意味着 Java 主系统可以不直接嵌死所有工具实现，而是通过 gateway 接 SSE、stdio、bridge、fastMCP、数据库适配器。

8. MCP 交互流程  
简化讲就是：
   - 发现 capability
   - 选择 gateway
   - 标准化参数
   - 发起 invoke
   - 处理超时/重试/fallback
   - 记录审计
当前项目的 `McpGatewayService` 就是这个流程的统一入口。

9. Agent 如何与 MCP Server 通信  
当前项目支持多种网关：
   - bridge
   - SSE
   - stdio
   - fastMCP
   - 数据库 MCP adapter
所以不是一种固定通信方式，而是统一抽象成 `McpCapabilityGateway`。

10. MCP 暴露了哪些工具接口  
从当前代码看，重点是数据库和检索相关能力，例如 Neo4j、Milvus，以及通用 capability 调用。  
项目里还保留 stub gateway，便于开发和失败兜底。

11. MCP 调用失败或误调如何处理  
当前实现已经比较完整：
   - 参数先 normalize
   - gateway 选择失败可以 fallback stub
   - 调用异常按 retryable 判断是否重试
   - 最终统一记录 audit
   - 返回结构化结果，不让调用链整体崩溃

12. RAG + MCP 怎么做  
这个方向我会这样解释：  
RAG 负责找知识，MCP 负责调能力。  
当检索命中不足、需要结构化查询或数据库能力时，不一定继续靠语义检索，可以通过 MCP 去查图库、图库、Neo4j、Milvus 或其他外部系统。  
当前项目已经把数据库 MCP adapter 骨架搭出来了，本质是在把“知识检索”和“外部能力调用”打通。

13. 并行化意图识别是什么  
严格说，当前项目主链路更偏分层识别，不是完全并行意图分类。  
但这个概念我理解为：把域识别、规则识别、历史上下文抽取、候选任务召回拆成可并行组件，然后在融合层决策。  
为什么要并行：减少串行等待，提高 top-level routing 的鲁棒性。  
如果后续扩到 100+ 工具，我会把“工具候选召回”和“业务域分类”并行跑，而不是一层一层堵住。

### 7. 多 Agent 协作

1. 对 A2A 或 Multi-Agent 的理解  
我不把多 Agent 看成“多开几个 prompt”，而是看成多个有边界职责的执行体，通过消息协议协作。  
当前项目已有 `A2ABus`、`A2AMessage`、`A2AIntent`、`A2AStatus`。这已经具备轻量多 Agent 协作骨架。

2. A2A 与 MCP 区别  
   - A2A 是 Agent 和 Agent 之间协作。
   - MCP 是 Agent 和 Tool/Capability 之间协作。
前者解决任务分工，后者解决能力接入。

3. 多 Agent 中记忆如何隔离与共享  
我的原则是：
   - 原始会话记忆按 session 隔离。
   - 长期用户记忆按 user 归属。
   - 共享的是结构化结果，不共享整个 prompt 上下文。
这能避免工具和代理之间相互污染。  
当前项目通过任务 payload、context、A2A metadata 来做边界控制，不是直接把全量聊天历史广播给所有 Agent。

### 8. 大模型原理与 Attention

1. LLM 底层原理，输入给模型的是什么  
输入本质上是 token 序列，先嵌入成向量，再经过多层 Transformer。  
模型看到的不是“字符串意义”，而是 token embedding 加位置编码后的张量。

2. self-attention、QKV、同 token 向量是否一样  
同一个 token 在输入 embedding 上可以相同，但进入每层后，经过不同线性投影，会得到不同的 Q、K、V。  
Q 表示“我现在想找什么”，K 表示“我有什么可被匹配的信息”，V 表示“真正要被聚合的信息内容”。

3. 为什么 attention 能建模长距离关系  
因为任意 token 都可以直接和其他 token 建联系，不像 RNN 必须逐步传递。  
所以它天然擅长远距离依赖，但代价是复杂度高。

4. 为什么需要 multi-head  
一个头很难同时学会语法、指代、位置、逻辑、实体关系。  
多头可以并行捕获不同模式，所以它本质上是多视角动态加权。

5. 为什么同一个 token 的 Q、K、V 不同  
因为它们承担的语义角色不同，来自不同投影矩阵。  
同一个人对外问问题、被别人索引、输出信息，不会是同一表示。

6. 长上下文怎么优化  
工程上常用：
   - 摘要压缩
   - 分层检索
   - 分块生成
   - sliding window
   - KV cache
   - 稀疏注意力或线性注意力模型
当前项目主要用的是摘要压缩、检索式上下文注入、分阶段执行，而不是把所有历史一次塞满。

7. Attention 稀释为什么伤指令遵循  
因为高价值指令在超长上下文里会被大量低价值 token 稀释，模型对关键约束的注意力分配下降。  
这也是为什么 Agent 场景里必须做上下文工程，而不是迷信超长窗口。

8. 二次复杂度在 Agent 工程里会带来什么问题  
   - 首包慢
   - 成本高
   - 上下文越长越不稳
   - 工具观察结果不断拼回去后，后续推理质量下降
所以项目里才有上下文压缩、证据截断、流式首包探测。

9. LoRA 和 QLoRA  
LoRA 是冻结原模型，只训练低秩增量矩阵。  
QLoRA 是把基座模型量化后再做 LoRA 训练，核心是减少显存占用。  
QLoRA 通过低比特量化基座、分页优化器、低秩适配，能在较小显存上训练较大模型。

10. 多模态大模型结构  
通常是：
   - 文本编码器/解码器
   - 视觉编码器，例如 ViT
   - 中间对齐层或 projector
   - 跨模态融合层
这个项目目前更多是“视觉摘要 + 图文检索增强”，不是自己训练一个原生 VLM。

### 9. 模型幻觉治理

1. 如何衡量 Planning 能力和 Hallucination  
我会拆指标：
   - Planning 成功率：任务完成率、工具序列正确率、平均步骤数、澄清命中率、回退率
   - Hallucination：引用不存在证据比例、事实冲突率、答案忠实度、非法工具调用率
自动评估框架可以用 RAGAS、自定义 judge、流程回放和 trace 审核结合。

2. 为什么仅靠 RAG 解决不了幻觉  
因为幻觉来源不只在“没知识”，还包括：
   - 路由错
   - 工具错
   - 上下文污染
   - 规划错误
   - 模型过度补全
在电商这种强事实场景，更不能只靠 RAG，必须加：
   - 结构化数据源优先
   - 价格库存走实时接口
   - 高风险字段强约束输出
   - 不确定即澄清/拒答

3. 幻觉常见原因和降低方法  
常见原因：
   - 检索不到
   - 检索错
   - 上下文太长
   - prompt 含糊
   - 工具结果未校验
   - 模型倾向补全
降低方法：
   - 证据约束
   - 结构化输出
   - 工具白名单
   - 低置信澄清
   - 多阶段验证
   - 高风险字段直接查库而不是生成

4. 当前项目怎么处理胡编乱造  
   - 检索阶段做 evidence 编号。
   - 生成阶段只允许引用编号证据。
   - 低质量召回时 Web fallback 或直接降级。
   - 高风险工具调用走参数校验和审计。
   - 评测层用 retrieval eval 和 quality eval 持续回归。
这套组合比单点“加一句不要幻觉”有效得多。

### 10. 上下文工程与长文本处理

1. rerank 后一般返回几个块  
当前 `buildKnowledgePacket` 主入口里默认混合检索 topK 是 5。  
但这里要注意，真正进生成上下文的不一定严格等于 5 个原始块，还会受父子回填、图片合并、截断长度影响。

2. 为什么是这个数量  
不是因为 5 神奇，而是当前场景下 5 往往能在“证据覆盖”和“噪音控制”之间取得平衡。  
块数太少，证据不够；块数太多，注意力被稀释。

3. rerank 后 topK 截断怎么做  
先召回，再融合，再重排，最后截到有限高价值块。  
更关键的是字符级截断和 evidence 目录控制，而不是只盯 topK 数值。

4. 为什么是这个值，有没有验证  
应该通过评测验证，不应该凭经验永远固定。  
这个项目已经具备做这类实验的条件，我建议把不同 topK 和 chunk 组合跑成参数矩阵，而不是一次只盯一个值。

5. 上下文长度不够或过长怎么处理  
当前项目做法：
   - 会话历史用摘要压缩。
   - 检索上下文按长度截断。
   - profile snapshot 截断。
   - imageContext 截断。
核心原则是：优先保留最近约束、最强证据、结构化摘要。

6. 上下文工程怎么设计  
我设计时会分三类上下文：
   - 身份上下文：用户画像、角色设定、系统规则
   - 任务上下文：当前目标、澄清结果、执行阶段
   - 证据上下文：检索片段、图片摘要、结构化事实
不同上下文不能混写，否则模型会把规则、历史、证据混成一锅。

7. 上下文拼接结构  
当前项目典型结构是：
   - `【用户画像】`
   - `【会话摘要】`
   - `【近期对话】`
   - `【知识上下文】`
   - `【图片上下文】`
   - `【证据目录】`
这是比较标准的分区式拼接。

8. 如何避免上下文过长导致性能下降  
   - 只保留相关块
   - 历史摘要化
   - 证据编号化
   - 无关图片和无关对话剔除
   - 对不同任务用不同模板

9. 当前上下文如何处理  
当前项目里最关键的是 `ChatContextCompressor`。  
消息超过阈值后，会保留最近若干轮原文，把更早内容压成 summary，再把稳定偏好抽到长期记忆。  
这比一直堆历史消息要成熟很多。

10. 优化思路  
如果继续优化，我会做：
   - 细粒度消息打标签，而不是统一摘要
   - 把“约束”“事实”“偏好”“未完成任务”拆分存储
   - 引入 retrieval-on-history，而不是只靠 summary

11. 如何减少无关上下文干扰  
   - 先路由，再检索
   - 先压缩，再拼接
   - 历史和证据分区
   - 对不同 scenario 使用不同 handler

12. 如何在窗口不足时保留初始 Attention Sink  
做法不是死保全文，而是保留“初始任务锚点”：
   - 首轮目标摘要
   - 关键约束列表
   - 已完成/未完成步骤
   - 关键证据引用
这比机械保留第一轮原文更有效。

13. 如何缓解信息遗忘  
   - 会话摘要
   - 长期记忆抽取
   - 状态机
   - 按主题做 history routing
   - 必要时二次澄清

14. 摘要容易丢细节，怎么处理  
不要只保留一份自然语言摘要。  
正确做法是“摘要 + 结构化关键事实 + 可回查原文锚点”。  
当前项目已经在往这个方向走，但还可以继续加强。

15. 任务远大于单次 Token 限制，如何支持断点继续  
要把任务变成可恢复状态机：
   - 每一步有 taskId
   - 保存阶段结果
   - 输出增量事件
   - 支持 cancel / retry / resume
当前项目在流式面试链路已实现 `streamTaskId` 和取消闭环，但还没有真正持久化的生成 checkpoint。  
所以我的回答会很实事求是：现在支持取消与部分会话恢复，不是完整的 token-level checkpoint resume。

### 11. 系统设计与性能优化

1. 3 个以上工具调用且高频请求，怎么压低端到端延迟  
   - 多级路由，先过滤掉不需要工具的请求
   - 工具候选召回，而不是全量枚举
   - 可并行的工具并行
   - 首包探测 + 熔断降级
   - 缓存 query rewrite、工具元数据、静态结果
   - 流式返回中间进度

2. 工具超过 100 个怎么做路由  
我会做三层：
   - 规则黑白名单
   - 向量/关键词检索召回候选工具
   - 小模型在候选集内二次选择
避免让大模型在 100+ 工具描述上直接做全量选择。

3. 如何避免循环调用/思维死循环  
   - 限制最大步数
   - 相同工具相同参数去重
   - 增加“连续无收益调用”检测
   - 状态机强约束
   - 必要时强制澄清或降级回答
我倾向于把“停止条件”写进执行框架，而不是期待模型自己懂得停。

4. GraphRAG 对复杂关联查询的优势  
优势在于：
   - 结构关系显式
   - 能处理“谁依赖谁”“上下游影响”“相关概念链路”这类问题
   - 对多跳关系和概念扩展更稳
当前项目中的图谱检索已经在把概念名、类型、描述拼成自然语言上下文，而不是只返回节点名。

5. 一千条数据求和怎么设计  
如果只是求和，不需要 LLM。  
工程上就是：
   - 输入校验
   - 分批读取
   - 流式聚合
   - 输出结果和校验日志
LLM 只适合解释任务，不适合替代确定性计算。

6. 百万级怎么处理  
   - 分片
   - 流式处理
   - 批处理作业
   - 幂等和 checkpoint
   - 结果汇总
千万不要把百万条记录塞给 Agent 自己“慢慢算”。

7. 如何保证效率和稳定性  
   - 把确定性计算与生成式能力分层
   - 关键路径减少模型调用次数
   - 超时、重试、熔断
   - 幂等和审计
   - 观测与告警

8. 智能导购助手 Agent 怎么设计  
四层我会这样拆：
   - 感知层：理解用户预算、品牌偏好、使用场景、模糊词
   - 规划层：决定先澄清还是先召回商品
   - 记忆层：长期偏好、价格敏感度、购买历史
   - 执行层：查商品库、查库存、查优惠、生成比较结论
分布式下再加：
   - 用户画像服务
   - 商品检索服务
   - 实时价格库存服务
   - 对话编排服务
强事实字段必须走结构化接口，不能让模型生成。

9. 模糊需求怎么精准理解  
核心不是“猜”，而是做模糊信息分解：
   - 识别约束缺口
   - 提取已知偏好
   - 判断是否足以执行
   - 不足时主动澄清
比如“想买个适合出差的轻薄本”，至少要补预算、系统偏好、是否游戏、续航优先级。

10. 主动澄清怎么设计  
我会看三类信号：
   - 风险：猜错代价高不高
   - 置信：top1 和 top2 差距够不够大
   - 历史：用户偏好是否足够稳定
高风险低置信必须澄清；低风险高置信可以直接做；中间态可以给一个默认假设并允许用户纠正。  
当前项目的意图树已经实现了低置信澄清，这个思路是对的。

### 12. 安全与防护机制

1. Prompt 注入如何防御  
不能只靠一条提示词。  
要做：
   - 系统指令和用户输入分区
   - 外部内容标记为不可信证据
   - 工具白名单
   - 敏感操作二次确认
   - 输出后校验

2. 有没有输入过滤和规则校验  
有基础能力。代码里有 `InputSanitizer`、CORS、安全配置、参数校验。  
但如果面向生产，我还会再加 prompt injection 检测、敏感关键词策略和工具权限矩阵。

3. 工具调用安全控制怎么做  
   - capability 白名单
   - 参数标准化
   - 超时/重试限制
   - 审计日志
   - fallback 而不是穿透到危险系统

4. 如何限制模型调用敏感接口  
不要让模型直接持有敏感接口描述。  
应该通过后端代理层做授权判断，只开放允许的 capability。

5. 是否有权限控制或白名单机制  
当前项目已经有一定基础，但还可以继续增强到租户级、角色级、工具级权限控制。  
对外表达时我会说：现阶段重点做了调用链治理和安全兜底，细粒度 RBAC 还可以继续补强。

### 13. AI 框架选型与使用

1. 为什么选 Spring AI  
因为当前项目主体就是 Java Spring Boot，Spring AI 更适合直接接入现有工程体系：
   - 依赖注入自然
   - 和 Spring 生态整合顺手
   - 便于和数据库、Web、配置体系统一

2. Spring AI 的优势  
   - 对 Java 团队友好
   - 易于接已有企业后端
   - 模型接入和 Bean 管理统一
   - 更适合做强工程化、弱脚本化的系统

3. 如何设计多模型支持架构  
当前项目已经做了 `RoutingChatService + ModelSelector + ModelHealthStore + RoutingExecutionTemplate`。  
核心思想是：
   - 模型候选配置化
   - 路由类型区分，例如意图模型、思考模型、默认模型
   - 首包探测
   - 熔断降级
   - 指标观测

4. 多租户下模型切换是否支持热更新  
当前项目已经支持数据库候选模型和运行时观测，但严格租户级独立热切换还没完全做完。  
如果要做，需要把模型候选、API Key、熔断状态都引入 tenant scope。

5. Agent 的记忆、工具调用、知识检索如何实现  
   - 记忆：消息表、会话摘要、长期记忆表、学习画像
   - 工具：MCP gateway + 受控 service 层
   - 知识检索：RAGService 统一入口，混合检索 + 父子索引 + 图片检索

6. 用过哪些 Agent 框架，怎么选  
我对 LangChain/LangGraph、Spring AI、MCP 生态都了解。  
选型标准不是流行度，而是：
   - 是否适合当前技术栈
   - 是否能承接现有工程
   - 是否容易做观测、权限、测试、部署

7. 对 LangChain 等的理解  
LangChain/LangGraph 适合快速原型、复杂图式编排和 Python 生态。  
但当前项目是 Java 主体、业务系统很重，所以我更倾向 Spring AI + 自研编排。

8. 为什么选 DeepSeek 和千问/其他候选  
核心标准通常是：
   - 中文能力
   - 成本
   - 推理能力
   - 时延
   - 可用性
   - 接口兼容性
我不会把模型选型讲成“个人喜好”，而是讲成不同任务路由下的能力-成本权衡。

9. “成本优先”策略由谁提出  
如果面试官问这个，我会直接说：这是我基于项目阶段、调用量和预算约束给出的工程策略。  
不是框架规定，也不是模型厂商要求，而是架构 owner 需要做的取舍。

### 14. SSE 与前后端通信

1. SSE 前后端如何交互  
当前项目后端暴露 `text/event-stream` 接口，前端用流读取器持续消费事件。  
面试链路有 `/api/interview/stream/start|answer|report|stop`，聊天链路有 `/api/chat/sessions/{sessionId}/stream`。

2. 后端推送什么数据格式  
事件名 + JSON payload。  
典型事件包括：
   - `meta`
   - `progress`
   - `message`
   - `finish`
   - `cancel`
   - `error`
   - `done`

3. 工具调用时 SSE 事件通常包含哪些字段  
如果是步骤级事件，我会包含：
   - `traceId`
   - `streamTaskId`
   - `stage`
   - `status`
   - `toolName`
   - `inputSummary`
   - `outputSummary`
   - `percent`
   - `timestamp`
当前项目面试流里重点已有 `action / traceId / stage / label / percent / channel / delta / result` 这些字段。

4. 步骤级 SSE 和普通 Token 流的本质差异  
普通 Token 流只解决“字一个个出来”。  
步骤级 SSE 解决“系统现在在做什么”。  
它能表达准备中、检索中、生成反馈中、已取消、已完成，这对 Agent 场景更重要，因为用户关心的不只是 token，还关心阶段状态。

5. 通信中途断连如何恢复  
当前项目已经支持：
   - `streamTaskId`
   - 任务取消
   - 会话本身持久化
但严格来说，还没有做到“断线后从上一个 token 或上一个步骤自动续传”。  
所以我会明确说明：现在更偏“可取消、可重试、会话可恢复”，不是完整的 SSE checkpoint replay。

6. Checkpoint 存储为什么选择  
以当前项目现状，真正持久化的是 session 和消息，不是 SSE token checkpoint。  
这是合理的第一阶段选择，因为：
   - 业务恢复比 token 恢复更重要
   - token 级 checkpoint 成本高、实现复杂
   - 大多数场景重放一个步骤比续 token 更实用
如果后续要做，我会把 checkpoint 落到任务状态表，而不是只放内存。

### 18. 多模态功能实现

1. 系统如何实现图像识别  
当前项目走的是两段式：
   - 视觉模型先做图片摘要/embedding
   - 再把图片作为知识对象参与检索
对应组件包括 `ImageService`、`ImageMetadataCollector`、`ImageEmbeddingService`、`ImageIndexService`。

2. 为什么不直接用多模态大模型  
因为知识库场景里，很多时候需要的是“可检索、可关联、可复用的图片知识”，不是单次看图问答。  
直接把每次图片都交给多模态大模型，成本高，也不利于沉淀结构化关联。

3. 没有前端界面时，本地图片怎么传到后端  
可以通过文件上传接口或入库同步流程。  
在知识库场景下，更常见的是 Markdown 里引用本地图片，入库时自动解析并建立关系。

4. 多模态用户信息怎么存储和使用  
图像本身的元数据、摘要、向量、文本块关联都会落存储。  
真正给生成模型用时，不是把原图二进制塞进去，而是给“图片摘要 + 与文本关系 + 命中原因”。

### 15. 项目相关问题

1. Agent 项目背景  
这是一个面向技术面试复盘、知识检索、刷题训练、RAG 评测和多模态检索的本地 AI 系统。  
它不是单纯聊天机器人，而是围绕“面试学习闭环”做出来的综合平台。

2. 演示项目实现细节怎么讲  
我会按一条主链路讲：
   - 用户输入
   - `TaskRouterAgent` 路由
   - 命中面试/刷题/知识问答 Handler
   - 需要知识时走 `RAGService`
   - 模型调用走 `RoutingChatService`
   - 过程通过 SSE 输出
   - 结果沉淀到会话、画像、评测和观测
这条链清晰，比逐个类讲更容易让面试官理解。

3. AI 辅助开发经验  
我的原则是：AI 适合提速，不适合替代架构判断。  
适合让 AI 做：
   - 初版代码骨架
   - 测试样例
   - 文档整理
   - 重复性重构
但关键设计、边界判断、验收口径必须由工程 owner 收口。

4. AI 贡献代码占比  
我不建议给死数字。  
更稳妥的说法是：AI 参与了大量编码提效和局部实现，但核心架构设计、链路裁剪、验收与重构收口由我主导。  
这是更专业的表述。

5. Session 和 User ID 如何绑定  
当前项目通过 `UserIdentityResolver` 解析用户身份，再由会话服务创建/查询 session。  
所以 session 是用户域下的会话对象，不是全局裸 session。

6. 提示词模板如何设计和迭代，怎么判断变好  
当前项目已经把 Prompt 去文件化，统一收口到数据库模板。  
迭代方法应该是：
   - 明确任务目标和输出 schema
   - 配 Few-shot
   - 上评测集
   - 看错误分布
   - 小步迭代
判断是否变好，不看主观感觉，看：
   - 任务成功率
   - 结构化输出稳定性
   - 幻觉率
   - 评测指标
   - 人工 spot check

7. 怎么量化 Agent 性能  
我会分四类指标：
   - 业务指标：任务完成率、澄清成功率、面试/刷题闭环率
   - 模型指标：成功率、熔断率、首包时延、fallback 率
   - RAG 指标：Recall@k、MRR、faithfulness
   - 系统指标：P95 延迟、错误率、取消率、SSE 完成率
Agent 性能从来不是单一“回答像不像人”，而是整条任务链路的综合质量。

## 可作为面试收口的总结表述

如果面试官只给我 1 到 2 分钟，我会这样总结这个项目：  
这是一个以技术面试学习闭环为核心的 AI Agent 系统，后端是 Spring Boot，前端是 Vue 3。系统通过 `TaskRouterAgent` 做分层意图路由，通过 `InterviewOrchestratorAgent / KnowledgeQaAgent / CodingPracticeAgent` 分担不同任务；知识侧采用父子索引、向量检索、词法检索、GraphRAG 和图文联合检索；模型侧通过 `RoutingChatService` 做多模型路由、首包探测、熔断和降级；会话侧通过上下文压缩、长期记忆提取和学习画像实现多轮承接；观测侧通过 RAG Trace、检索评测、质量评测和 SSE 步骤流实现可诊断与可优化。  

如果一句话概括我的设计取向，就是：我不追求“自由发挥型 Agent”，而追求“可控、可观测、可评测、可恢复的工程化 Agent 系统”。
